Did the team implement Deployment Validation and automated Rollback Capabilities as a final quality gate?
 
>> The deployment validation was performed for every deployment, albeit manually.  For each deployment, once the deployment was completed, a two-step validation procedure was executed.  

1) On the AWS Elastic Beanstalk application page, verify that the current deployment to the target environment matches the latest build, also checking that the Health of the application is Ok.
2) Then, direct a browser to the application site and conduct smoke test.  
>> The rollback procedure is as follows.  It’s not completely automated, but it is just a push of a button.
1) On the AWS Elastic Beanstalk application page, navigate to the Application versions.
2) Select the version of the application to roll back to and select Deploy from the Actions menu.
How complete was the Continuous Integration Implementation?  

>> Continuous Integration was implemented for all branches of code.  A single Bitbucket pipeline would automatically trigger to generate a build of the source regardless of what branch the source changes were committed to.  The resulting build artifacts were uploaded to an S3 bucket in AWS for potential deployment.  

>> For builds that were generated from the dev or master branch, the CI implementation would further trigger a CD process to automatically deploy the generated build to the Dev environment for dev-branch builds and to the Prod environment for master-branch builds.

‍What sort of Deployment methodology was implemented?  

>> The deployment of the web app was automated as a Bitbucket pipeline.  Deployments to the Dev and Prod environments were triggered to occur automatically following successful builds from the dev or master branches of source.  

The Bitbucket pipeline specifically leveraged the aws-elasticbeanstalk-deploy pipe to deploy build artifacts to the target environment of our Elastic Beanstalk application.  Here is the reference page for the aws-elasticbeanstalk-deploy pipe.        

‍What type of Source Control Branching methodology was used?  

>> The branching methodology that we used was a derivative of the Gitflow Workflow branch model.  Here is a description of the Gitflow Workflow branch model.  

The only difference between the branching methodology that we used, and the Gitflow Workflow is that we did not utilize release branches in our model.  We did not use Release branches, because we anticipated only having a single release to Production at the end of the Hackathon.  Thus, instead of forking a release branch from the develop branch and then merging the release branch to the master branch, we eliminated the Release branch, and simply merged the develop branch with the master branch when we were ready to release to Production.  

All our development was done on feature branches.  Using feature branches allowed us to isolate the development work related to stories on specific branches.  Once a story was code-complete, we would issue a pull request to merge the feature branch to the develop branch to integrate the new functionality with the rest of the code.  The newly integrated code would be built and deployed to the Dev environment for integration-level testing.  

At the end of the Hackathon, the develop branch was merged with the master branch.  A build from the master branch was then generated and that build was then deployed to the Production environment of our Elastic Beanstalk application.
